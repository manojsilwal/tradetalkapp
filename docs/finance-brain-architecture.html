Request flow: /decision-terminal

  1. Cache check — backend/verdict_cache.py returns cached payload for the trading day (unless force=1).
  2. Parallel fetch — Polymarket, yfinance valuation snapshot, live quote, scorecard embed, momentum model inputs.
  3. Swarm trace — backend/agents.py: macro snapshot → 4 factor agent pairs (short interest, social sentiment, Polymarket, fundamentals) in parallel. Optional LLM for ambiguous cases.
  4. Debate — backend/debate_agents.py: 5 specialist LLM agents (bull, bear, macro, value, momentum) + moderator. Each agent queries RAG then calls llm_client.generate_argument().
  5. Assembly — backend/decision_terminal.py builds 4-panel payload: valuation (DCF heuristics), momentum (deterministic math), quality/scorecard, agent roadmap (LLM).
  6. Emit — Decision ledger + CORAL hub handoff events.

Section 02

Why It Is Slow & Non-Deterministic

Every uncached analysis request is a cold start across the entire data + reasoning stack. The verdict cache helps repeat views of the same ticker on the same trading day, but first-load and multi-ticker workflows pay full cost.

❌ Today: Request-Time Everything

  • 15+ connector calls with 45–300s TTL — no durable feature store
  • 6–8 sequential-ish LLM calls (5–30s each)
  • Scores vary run-to-run (LLM non-determinism)
  • No model version or as-of date on numerical signals
  • Cannot backtest the "brain" — there is no brain
  • Cloud IP blocks cause yfinance failures → FinCrawler fallback chain
  • High API cost per user session
  • Total latency: often 30–90+ seconds on cache miss

✓ Target: Precomputed Brain + Explain Layer

  • Daily batch: ingest → features → predict → store
  • API serves cached prediction JSON in <200ms
  • LLM reads structured scores — does not invent numbers
  • Every output stamped with model version + as-of date
  • Walk-forward backtested before deployment
  • Hot/warm/cold tiering with provider fallback
  • LLM cost only for explanation/debate (optional)
  • Total latency: <2s for scores, +5–10s for memo

Bottleneck breakdown

Data fetch fan-out

Target API contract: analyze a ticker (dynamic — base + live)

The response is never a frozen score. It carries the precomputed base block, the stored anchors (intrinsic value, base price), and a live block produced on-demand by the Reflex layer (Section 03B), plus a status and a base→current waterfall. Freshness lowers confidence_score, not the score.

The Brain is model-agnostic — best validated model wins

Design decision: do not fetishize the neural net. On tabular, low-signal-to-noise cross-sectional finance data, gradient boosting (XGBoost / LightGBM) usually beats an MLP. The "brain" is whichever model wins the validation gate — the architecture stays swappable behind the inference contract. TradeTalk already has a model-swap replay harness to support this. 

Candidate ladder (all compete, all versioned):
  1. Rule-based composite score   (transparent floor — must beat random)
  2. Logistic regression          (linear baseline)
  3. Random forest                (non-linear baseline)
  4. Gradient boosting (XGB/LGBM) (likely production winner on tabular data)
  5. FinancialRankingNet (MLP)    (optional head)
  6. TimesFM                      (ACTIVE time-series head, reuses backend/predictor/)

Winner = highest precision@topK AND survives the validation gate below.
TimesFM is wired in, not a footnote
The existing TimesFM predictor (backend/predictor/ + the tradetalk-timesfm/ microservice) is the brain's forward-looking time-series head. It produces probabilistic price-path quantiles (q10/q50/q90 in USD per horizon). backend/brain/timeseries.py converts the 63d band + the current price into two model features the cross-sectional classifier consumes (tsfm_expected_return, tsfm_band_width) plus a UI-facing forecast block, and adds a dedicated "timeseries" signal group. The TimesFM quantiles are stored as absolute-USD anchors on the snapshot, so the Reflex layer recomputes the implied forward return against a live price without re-running TimesFM — exactly like the DCF anchor. backend/brain/timesfm_adapter.py is the lazy async bridge to run_predictor_forecast() (offline-safe: the brain imports with zero predictor deps).

Validation gate — guards against leakage & overfitting

63-day forward labels on daily data are heavily overlapping and autocorrelated. A naive walk-forward random split leaks the future into training. Because we will try many models/features, backtest return alone is noise. A model ships only if it clears all gates:

  • Purged + embargoed CV — remove overlapping samples around each test fold (López de Prado), not just a date split.
  • Beat rule + ML baselines — on precision@topK and calibration (Brier), not just AUC. Realistic AUC for this task is ~0.52–0.58; ranking quality matters more than raw accuracy.
  • Deflated Sharpe / multiple-testing correction — discount the backtest for the number of trials to avoid selecting an overfit winner.
  • Paper-trade forward test — run the candidate live (read-only) in the existing paper portfolio before any prediction is trusted in the UI.

Section 03B

Reflex Layer — Dynamic Brain, Not a Frozen Score

Precomputation makes us fast; the Reflex layer makes precomputation safe. We never store only {score, verdict}. We store score components + valuation anchors + the base feature row, then compute a live-adjusted view when the user asks. The mechanism is not hand-tuned point deltas — it re-derives the price-sensitive features from the stored anchors and re-runs the model (one O(1) forward pass), so momentum and valuation move consistently.

1. Memory Layer
Raw & processed data, prices, filings, events, features. (Plan B pipeline.)
2. Brain Layer (nightly, slow)
DCF/intrinsic, quality/valuation/momentum/risk, model probability, base ranking. Stored as a snapshot with anchors.
3. Reflex Layer (on-demand, fast)
Live price/sector/events/sentiment → recompute price features → re-run model → current view + waterfall. The only part the naive design skips.
flowchart LR

The headline example (why frozen scores are dangerous)

Yesterday:  base_price $125,  intrinsic_mid $160  ->  DCF upside +28%,  score 82, BUY
Today:      stock +25% to $156.25  (intrinsic UNCHANGED at $160)

Frozen design:   still shows 82 / BUY                      ❌ stale, valuation gone
Reflex design:   intrinsic fixed -> DCF upside = 160/156.25 - 1 = +2.4%
                 valuation signal falls, model re-runs on updated features
                 -> live verdict "Buy / Watch — valuation less attractive after the move"  ✓

Flaws in the naive "add deltas" design — and how the Reflex layer fixes them

Naive flaw Fix in this design
Adding "valuation -14 points" mixes units with an ML scoreRe-derive features from anchors, re-run the model, re-aggregate with the same functions; deltas are display-only
A price move only routed into valuation (double-counts momentum)One live price updates returns, MAs, valuation ratios & relative strength together
Intrinsic value treated as fixed even when rates/guidance moveMaterial events & rate moves invalidate the anchor and force a fresh brain run
Event deltas as magic constants (+4 earnings)Events classified; material ones invalidate, they do not nudge
Freshness penalty subtracted from the scoreFreshness lowers confidence only; the score stays the best estimate
Verdict flips BUY/HOLD on noise near thresholdsHysteresis bands + a component waterfall, not a bare flipped label
A 2:1 split looks like a -50% crashSplit-ratio adjustment + sanity bounds on the live price
A market-wide selloff changes a sector-relative callIdiosyncratic = total - sector; market-wide moves leave the relative thesis ~unchanged
Ledger grades the base decision, not what the user sawReflex emits the live-adjusted decision (base + reflex components) to the ledger
Per-request LLM sentiment reintroduces latencyLive sentiment is a cheap cached signal (TTL); reflex math is O(1); LLM stays explanation-only

Data Completeness Reviewer + rate-limit-aware backfill loader

Design decision (free-data backbone): stay on free/free-tier providers, but never trust a single one. A completeness reviewer sits between providers and the canonical store and keeps switching sources until the data for a ticker/date-range is whole. A rate-limit-aware loader paces requests to each provider's implied quota and runs as a long-lived GCP job for hours/days until the historical backfill finishes — without ever blocking a live user request.

NEW Completeness Reviewer (the gate)

Only "complete-data-only" providers pass. The reviewer validates each normalized payload and re-routes on failure:

  • No gaps — continuous trading-day coverage across the requested window (no missing dates).
  • No missing fields — required columns present (e.g. adjusted_close non-null for features).
  • Sanity bounds — no negative volume, ratios within plausible ranges, monotonic dates.
  • Cross-provider reconciliation — flag/quarantine when two providers disagree beyond tolerance.
  • On fail → switch provider and merge partial results; if still incomplete, mark gaps and enqueue background backfill.
NEW Rate-limit-aware loader (the engine)

A resumable, quota-pacing backfill runner deployed on GCP:

  • Per-provider token bucket from provider_registry.rate_limit (e.g. Alpha Vantage 5/min, 25/day) — never exceed implied limits.
  • Long-running job — runs for hours/days on the always-free e2-micro Compute Engine VM (free-tier friendly for slow multi-day loads) rather than billed Cloud Run Jobs.
  • Resumable — checkpoint per ticker/date so restarts don't redo work (extends data_lake/.checkpoint.json).
  • Quota-aware scheduling — round-robin across providers + keys to maximize daily throughput.
  • Writes raw before canonical — store provider payload, then validated canonical rows.

Non-Negotiables & Definition of Done

These engineering rules apply to every phase. Violating any of them invalidates model trust.

Rule 01
No lookahead bias — features use only data available as of the feature date.
Rule 02
No unsupported LLM-generated financial numbers — scores come from tools/models only.
Rule 03
No model prediction without model_version in the response.
Rule 04
No prediction without as_of_date stamped on every output.
Rule 05
No production data source that violates terms of service.
Rule 06
No claim of certainty — use probabilities and confidence bands.
Rule 07
No model deployment (any architecture) without beating rule + ML baselines under purged/embargoed CV and a paper-trade forward test.
Rule 08
No frontend score without drill-down explanation and signal breakdown.
Rule 09
No backtest without transaction costs (default 10bps + 5bps slippage) and deflated-Sharpe / multiple-testing correction.
Rule 10
No agent memo without source fields cited from structured model output.
Rule 11
No data into the feature store without passing the completeness reviewer (no gaps, no missing required fields).
Rule 12
No external fetch that ignores a provider's implied rate limit — all backfill is quota-paced.
Rule 13
Every LLM number is verified against the structured payload; ungrounded numbers are flagged to contract_violations.
Rule 14
UI parity: no existing endpoint response shape changes; brain fields are additive only; cut over per-endpoint via shadow-mode with rollback.
Rule 15
Cloud portability: every cloud call goes through a port; DuckDB/Parquet + Supabase stay the source of truth; one Dockerfile + Terraform — no service used in a way that can't be swapped by config.

Section 10

Cloud Portability — Swap GCP ↔ AWS ↔ Azure

Use the same swappable-adapter philosophy you trust for LLMs and data providers, pointed at cloud infra. The stack is already ~90% portable because it runs on open primitives — a single Docker image, standard Postgres, Parquet + DuckDB, the Redis protocol, env-var config, and GitHub Actions cron. Only a few touchpoints are cloud-specific, and each sits behind a thin port chosen by a CLOUD_PROVIDER flag — exactly like the existing VECTOR_BACKEND / MCP_DATA_BACKEND switches.

Migration playbook — move with zero loss

  1. All state lives in portable stores — Parquet in object storage, Postgres rows, model artifacts as files, vector embeddings. Nothing critical is trapped in a proprietary service.
  2. Move the data — object-storage sync (gsutil → aws s3 sync / azcopy); for Postgres, either keep Supabase (cloud-neutral — a cloud move barely touches it) or pg_dump/restore.
  3. Stand up infra via Terraform — swap the provider module; same resource graph (container + bucket + db + cache + VM).
  4. Deploy the same container to the new runner; set CLOUD_PROVIDER + connection env vars.
  5. Parity check — run the shadow-mode/E2E suite against the new stack, then flip DNS/traffic. Rollback = point back.
  6. No retrain, no re-ingest — model artifacts + feature store + decision ledger carry over as files/rows; the brain wakes up identical.

Lock-in avoidance (do this from day one): keep DuckDB + Parquet as the source of truth (treat BigQuery as an optional accelerator), keep Supabase as the cloud-neutral data spine, use Terraform over gcloud-only scripts, ship one Dockerfile, and route every cloud call through a port. Do that and a GCP→AWS/Azure move is a config + data-sync exercise, not a rewrite.
